‚Źļ Here's my assessment of issue 486 ("Support fontSize and lineHeight for strong
  and em styles").

  TL;DR difficulty verdict

  Split the issue in two. fontSize on strong/em is a genuine help-wanted
  candidate ‚ÄĒ the plumbing is mechanical and there's a copy-paste precedent
  (code.fontSize already exists on both platforms). But your "28px bold inside a
  14px line" scenario exposes the real problem: both platforms hard-clamp the 
  line box to the block's fixed lineHeight, so oversized inline text overlaps
  neighboring lines today. Fixing that clamping is the actual work, and
  lineHeight as an inline prop is architecturally dubious on iOS and I'd
  recommend descoping it entirely.

  Notably, you can reproduce the broken behavior today without any code changes:
  set code: { fontSize: 28 } in a 14px paragraph ‚ÄĒ inline code already supports
  fontSize and goes through the exact same line-height clamping.

  What happens with test **testBold** test (14px / 28px)

  Android ‚ÄĒ StrongSpan is a MetricAffectingSpan, so making it set a larger
  textSize is a one-line change (StrongSpan.kt:26 currently hard-sets
  blockStyle.fontSize). The layout would then correctly measure the line using
  the tallest run's metrics, and all runs share the baseline. But then the
  custom LineHeightSpan (spans/LineHeightSpan.kt:23-25) runs and unconditionally
  squashes ascent/descent back to the fixed paragraph lineHeight:

  val leading = lineHeight - ((-fm.ascent) + fm.descent)  // negative when bold 
  is taller
  fm.ascent -= ceil(leading / 2.0f).toInt()               // ascent shrinks
  fm.descent += floor(leading / 2.0f).toInt()             // descent shrinks

  The 28px glyphs still draw at full size, so they bleed into the lines
  above/below (Android doesn't scale glyphs to the line box). The codebase
  already has a workaround pattern for exactly this:
  applyLineHeightSkippingImages in BlockSpacing.kt splits the LineHeightSpan
  around block images because "the fixed line height would re-clamp the metrics
  that ImageSpan expanded." The same strategy (or letting the metrics through
  when a taller inline run is present) would be needed for oversized strong/em.

  iOS ‚ÄĒ StrongRenderer.m resolves fonts per-character-range, so applying a
  bigger font is easy (superscript/subscript already scale fonts per range this
  way). But applyLineHeight (ParagraphStyleUtils.m:249) sets both
  minimumLineHeight and maximumLineHeight to the block value. With
  maximumLineHeight pinned, TextKit caps the line fragment and the 28px glyphs
  overlap adjacent lines, same as Android. There's a precedent escape hatch here
  too: math paragraphs already leave maximumLineHeight = 0 so the line can
  grow. A second, subtler iOS issue: applyBaselineOffset early-returns when the
  tallest font exceeds the paragraph lineHeight ‚ÄĒ so one bold word would
  silently disable vertical centering for the entire paragraph, shifting the
  14px text too.

  The input (out of scope, and cleanly so)

  The input has a separate, intentionally narrower style API ‚ÄĒ
  MarkdownTextInputStyle only allows strong.color / em.color (its own interface
  in EnrichedMarkdownTextInput.tsx, not the renderer's StrongStyle). On Android
  the input applies plain StyleSpan(Typeface.BOLD)
  (input/styles/BoldStyleHandler.kt), on iOS just bold traits + color. Adding
  fontSize there would drag in typing-attributes-at-span-boundary problems,
  cursor height, and toggle/merge logic. Since the types aren't shared, adding
  fontSize to the renderer doesn't force anything on the input ‚ÄĒ I'd explicitly
  state in the issue that the input is excluded.

  Other issues a contributor would hit

  - List markers: both platforms position bullets/checkboxes from font metrics
  at the line start (UnorderedListSpan.drawMarker uses paint.fontMetrics; iOS
  ListMarkerDrawer similar). A large bold run at the start of a list item shifts
  marker alignment.
  - iOS measurement cache: computeStyleFingerprint in MeasurementCache.h must
  hash the new strong.fontSize/em.fontSize fields (it already hashes
  code.fontSize at line¬†125) ‚ÄĒ forgetting this means stale measurements when the
  prop changes. Easy to miss.
  - Inline lineHeight semantics: on iOS lineHeight lives in NSParagraphStyle,
  which is effectively per-paragraph ‚ÄĒ an inline override can't be honored
  per-range. On Android a per-span LineHeightSpan would fight the paragraph one.
  CSS resolves this as "line box = max of runs on the line," which you get for
  free if you just let the line grow with fontSize. So lineHeight on strong/em
  adds a lot of complexity for a prop whose only sane meaning falls out of the
  fontSize fix anyway.
  - Web: trivial ‚ÄĒ strongStyle()/emphasisStyle() in src/web/styles.ts just need
  to emit the CSS props; browsers grow the line box natively.
  - Plus the routine surface: TS types, normalizeMarkdownStyle(.web).ts
  defaults, codegen spec, STYLES.md docs.

  Suggested framing for the help-wanted label

  Scope it as: add fontSize to strong/em (renderer only), and make lines grow 
  when an inline run exceeds the block lineHeight ‚ÄĒ pointing contributors at the
  code.fontSize plumbing precedent, the applyLineHeightSkippingImages / math
  maximumLineHeight = 0 escape hatches, and the measurement-fingerprint gotcha.
  Descope inline lineHeight (block lineHeight stays authoritative, line grows to
  fit) and the input. With that guidance it's a solid medium-difficulty
  help-wanted issue; without it, a contributor will ship the easy plumbing and
  the clipping will come back as a bug report.

‚Źļ Yes ‚ÄĒ and the architecture is already set up for it. Two facts make it
  feasible:

  1. The renderer runs on TextKit 1 with a custom layout manager.
  ENRMTextViewSetup.h:18 isa-swizzles the text view's layout manager to
  TextViewLayoutManager : NSLayoutManager. TextKit 1 exposes a per-line hook:
  NSLayoutManagerDelegate's layoutManager:shouldSetLineFragmentRect:lineFragment
  UsedRect:baselineOffset:inTextContainer:forGlyphRange: is called once per line
  fragment before it's committed, and lets you rewrite the fragment rect, used
  rect, and baseline offset for that line only. Subsequent lines stack on the
  modified fragment, so heights compose correctly.
  2. Measurement goes through the same layout manager
  (ensureLayoutForTextContainer + usedRectForTextContainer in
  ENRMUIKit.h:298-300, used by EnrichedMarkdownText.mm), so per-line adjustments
  made there are automatically reflected in the measured height ‚ÄĒ no separate
  measurement path to keep in sync. (The input measures via
  boundingRectWithSize, but that's out of scope anyway.)

  The design would invert the paragraph-level approach: keep minimumLineHeight
  as the floor, drop maximumLineHeight paragraph-wide when an oversized run is
  detected (as already planned), then in the per-line hook check whether this
  line's character range actually contains an oversized font ‚ÄĒ if not, clamp the
  fragment height back to the configured lineHeight and adjust the baseline
  offset accordingly. Net effect: only lines that truly contain big glyphs grow,
  matching Android's per-line chooseHeight behavior exactly.

  The costs, honestly stated:

  - Baseline math is fiddly. When you shrink a fragment you must recompute the
  baseline offset within it, or text visually jumps. This is exactly the class
  of work the existing TODO in ParagraphStyleUtils.m defers (per-line baseline
  centering, RN's enableIOSTextBaselineOffsetPerLine) ‚ÄĒ which is actually an
  argument for this hook: both follow-ups land in the same delegate method.
  - macOS is also supported by this library (TARGET_OS_OSX paths); AppKit's
  NSLayoutManager has the same delegate API but subtle metric differences, so it
  doubles the testing surface.
  - Small per-line cost of enumerating fonts in each line's range (cacheable per
  paragraph).

  My recommendation for the issue: keep the paragraph-level fix as the
  deliverable (the limitation only bites with compressed line heights combined
  with oversized inline runs ‚ÄĒ a rare corner), and add the per-line delegate
  approach to the "known limitations" section as the named follow-up mechanism
  rather than a vague "per-line centering is a separate follow-up". That
  upgrades the limitation from "we can't" to "we know exactly how, it's just
  scoped out". Want me to update the draft's iOS section to say that?
